# ML System Design Doc - [RU]
## Дизайн ML системы - Text To SQL
    
> ## Термины и пояснения
> - Итерация - это все работы, которые совершаются до старта очередного пилота  
> - БТ - бизнес-требования 
> - EDA - Exploratory Data Analysis - исследовательский анализ данных  
> - `Product Owner`,  `Data Scientist` - роли, которые заполняют соответствующие разделы 
> - В этом шаблоне роль `Data Scientist` совмещает в себе компетенции классического `Data Scientist` с упором на исследования и `ML Engineer` & `ML Ops` роли с акцентом на продуктивизацию моделей
> - Для вашей организации распределение ролей может быть уточнено в зависимости от операционной модели 

### 1. Цели и предпосылки 
#### 1.1. Зачем идем в разработку продукта?  

- Бизнес-цель - автоматическое определение сегмента целевой аудитории для персонализированных рекламных предложений продуктов Банка.
- Почему станет лучше, чем сейчас, от использования ML - сейчас уже контент рекламного предложения генерируется с помощью ML, но есть потребность в таргетированной рекламе для повышения Conversion Rate и других продуктовых метрик. Для этого надо для каждого нового рекламного предложения извлекать из внутренних баз данных айдишники клиентов, которым это предложение будет разослано. Сейчас эта задача выполняется вручную специальными людьми: они по рекламному предложению определяют нужный сегмент целевой аудитории (через грамотное составление критериев типа "возраст от 20 до 25, траты по картам в среднем от 50 тысяч в месяц" и т.д.) и пишут SQL скрипт для извлечения айдишников этого сегмента из внутренних баз данных.
- Что будем считать успехом итерации с точки зрения бизнеса - по описанию рекламного предложения генерируются текстовые критерии для определения целевой аудитории, а также по этоим требованиям извлекаются айдишники сегмента из внутренних базах данных.

#### 1.2. Бизнес-требования и ограничения  

- Краткое описание БТ: 
1. Система должна составлять критерии сегмента целевой аудитории исходя из содержания рекламного предложения (например, если рекламное предложение - "молодёжная кредитная карта", то полученный сегмент целевой аудитории должен состоять из молодёжи).
2. Экономия времени - среднее время формирования сегмента целевой аудитории должно снизиться минимум в 3 раза (люди всё ещё будут использоваться для формирования сегмента. Просто они не будут с нуля писать критерии сегмента и SQL скрипт, а будут изначально генерировать его через систему, а потом вносить исправления).
3. При экономии времени минимум в 3 раза, Conversion Rate не должен снизиться по сравнению с ручным составлением сегмента.
4. Система должна уметь извлекать сегмент с уже существующими критериями (указанными в текстовом виде), а не только извлекать сегмент аудитории с нуля.
5. Система должна уметь задействовать все доступные данные о клиенте из внутренних баз данных. Не должно быть такого, что система всегда генерирует SQL-скрипты, в которых только используется возраст клиента и его ежемесячные траты по карте.
- Бизнес-ограничения:
1. Система должна составлять критерии сегмента целевой аудитории исходя из доступных Банку данных о клиенте. То есть она не должна составлять критерии, которые нельзя вычислить через SQL скрипт по внутренним базам данных (например, если в базе есть данные только о возрасте, и доходе клиента, то нельзя состовлять критерий по семейному статусу клиента (в браке/один), так как таких данных нет в базе).
2. Один прогон системы (одна генерация SQL-скрипта) должен занимать максимум 5 минут.
3. В качестве LLM под капотом системы разрешается использовать либо open source модели, которые помещаются на доступные ресурсы (4 видеокарты A100 80GB), либо можно использовать по API модели серии GigaChat.
4. Если был выбран GigaChat, то за один прогон система должна потреблять максимум 200 тысяч токенов модели (без учета кэшированных токенов).
- Что мы ожидаем от конкретной итерации?
Ожидаем систему, которая умеет генерировать критерии сегмента целевой аудитории (в текстовом виде) по существующему рекламному предложению, далее по этим критериям она может генерировать SQL-скрипт для извлечения айдишников пользователей в сегменте. SQL-скрипт должен работать по схемам внутренних баз данных и также он должен завершаться успешно (без ошибок).
- Описание бизнес-процесса пилота, насколько это возможно - как именно мы будем использовать модель в существующем бизнес-процессе? Система будет использоваться как часть проекта Co-marketer (проект по автоматическому созданию и запуску рекламных кампаний). Co-marketer будет по запросу маркетолога генерировать контент для рекламного предложения, а Text to SQL система будет по контенту рекламного предложения (и возможно по доп. данным от маркетолога) генерировать сегмент целевой аудитории предложения через генерацию текстовых критериев сегмента и последующую генерацию SQL-скрипта, который по текстовым критериям извлекает айдишники людей этого сегмента из внутренних баз данных.
- Что считаем успешным пилотом? Критерии успеха и возможные пути развития проекта. Успешный пилот - система генерирует SQL-скрипты, которые завершаются без ошибок. Также на валидационном датасете система возвращает правильный айдишника сегмента целевой аудитории (пересечение между айдишниками пилота и таргет айдишниками в датасете должно быть >70%). Возможный путь развития - повышать процент пересечения айдишников при валидации системы.

#### 1.3. Что входит в скоуп проекта/итерации, что не входит   

- На закрытие каких БТ подписываемся в данной итерации: подписываемся на закрытие 1, 2 и 4 БТ.
- Что не будет закрыто - Под вопросом оставляем итоговый средний Conversion Rate для сгенерированного через Text to SQL систему сегмента. Допускаем, что он может быть в среднем на 1-5% ниже, чем был до использования системы. Также под вопросом остается пятое БТ. Вся схема внутренних баз данных может не поместится в контекст LLM и очень длинное описание схемы баз данных может портить качество сгенерированного SQL-скрипта. Поэтому скорее всего в данной итерации будет использоваться не вся схема БД, а только её часть.
- Описание результата с точки зрения качества кода и воспроизводимости решения - Код должен проходить CI проверки: линтер и форматтер. Также н валидационном датасете система должна генерировать правильные SQL-скрипты, то есть все сгенерированные скрипты должны завершаться без ошибок.
- Описание планируемого технического долга (что оставляем для дальнейшей продуктивизации) - В дальнейшем планируется улушчить систему, чтобы она генерировала SQL-скрипты основываясь на полной схеме БД, а не на ее части.

#### 1.4. Предпосылки решения  

- Описание всех общих предпосылок решения, используемых в системе – с обоснованием от запроса бизнеса: какие блоки данных используем, горизонт прогноза, гранулярность модели, и др.

Для валидации используем SQL-скрипты и айдишники, которые были составлены вручную по уже существующим рекламным предложениям. Не используем SFT, так как нет на это доступных ресурсов и достаточного количество данных. Вместо этого используем за основу pre-trained LLM. Саму систему строим используя промпт инжиниринг и структуры существующих мультиагентных Text-To-Code систем.

### 2. Методология `Data Scientist`     

#### 2.1. Постановка задачи  

- Что делаем с технической точки зрения: делаем мультиагентную систему на langgraph/langchain, которая может зайдествовать SQL-интерпретатор. Она на вход получает структурированные данные о новом рекламном предложении в виде json. Далее она с помощью LLM в текстовом виде генерирует критерии для сегмента целевой аудитории. После этого она по этим критериям генерирует SQL-скрипт для извлечения айдишников пользователей, которые удовлетворяют этим критериям. На выход система отдает критерии в текстовом виде, SQL-скрипт и результат SQL скрипта.

#### 2.2. Блок-схема решения  

- Блок-схема для бейзлайна и основного MVP с ключевыми этапами решения задачи: подготовка данных, построение прогнозных моделей, оптимизация, тестирование, закрытие технического долга, подготовка пилота, другое. `Data Scientist`  
- [Пример возможной блок схемы](https://github.com/IrinaGoloshchapova/ml_system_design_doc_ru/blob/main/product_schema.png?raw=true)
> Схема обязательно включает в себя архитектуру бейзлайна. Если бейзлайн и основной MVP отличаются несущественно, то это может быть одна блок-схема. Если значительно, то рисуем две: отдельно для бейзлайна, отдельно для основного MVP.  
> Если блок-схема **шаблонна** - т.е. её можно скопировать и применить к разным продуктам, то она **некорректна**. Блок-схема должна показывать схему решения для конкретной задачи, поставленной в части 1.    

#### 2.3. Этапы решения задачи `Data Scientist`  

- Для каждого этапа **по результатам EDA** описываем - **отдельно для бейзлайна** и **отдельно для основного MVP** - все про данные и технику решения максимально конкретно. Обозначаем необходимые вводные, технику предполагаемого решения и что ожидаем получить на выходе, чтобы перейти к следующему этапу.  
- Как правило, детальное и структурированное заполнение раздела `2.3` возможно только **по результатам EDA**.  
- Если описание в дизайн доке **шаблонно** - т.е. его можно скопировать и применить к разным продуктам, то оно **некорректно**. Дизайн док должен показывать схему решения для конкретной задачи, поставленной в части 1.  
    
> Примеры этапов:  
> - Этап 2 - Подготовка прогнозных моделей  
> - Этап 3 - Интерпретация моделей (согл. с заказчиком)  
> - Этап 4 - Интеграция бизнес правил для расчета бизнес-метрик качества модели  
> - Этап 5 - Подготовка инференса модели по итерациям    
> - Этап 6 - Интеграция бизнес правил  
> - Этап 7 - Разработка оптимизатора (выбор оптимальной итерации)  
> - Этап 8 - Подготовка финального отчета для бизнеса  

*Этап 1 - это обычно, подготовка данных.*  

В этом этапе должно быть следующее:  

- Данные и сущности, на которых будет обучаться ваша модель машинного обучения. Отдельная таблица для целевой переменной (либо целевых переменных разных этапов), отдельная таблица – для признаков.  
  
| Название данных  | Есть ли данные в компании (если да, название источника/витрин) | Требуемый ресурс для получения данных (какие роли нужны) | Проверено ли качество данных (да, нет) |
| ------------- | ------------- | ------------- | ------------- |
| Продажи | DATAMARTS_SALES_PER_DAY  | DE/DS | + |
| ...  | ...  | ... | ... |
 
- Краткое описание результата этапа - что должно быть на выходе: витрины данных, потоки данных, др.  
  
> Чаще всего заполнение раздела невозможно без EDA.

 *Этапы 2 и далее, помимо подготовки данных.*
 
Описание техники **для каждого этапа** должно включать описание **отдельно для MVP** и **отдельно для бейзлайна**:  

- Описание формирования выборки для обучения, тестирования и валидации. Выбор репрезентативных данных для экспериментов, обучения и подготовки пилота (от бизнес-цели и репрезентативности данных с технической точки зрения) `Data Scientist`    
- Горизонт, гранулярность, частоту необходимого пересчета прогнозных моделей `Data Scientist`   
- Определение целевой переменной, согласованное с бизнесом `Data Scientist`   
- Какие метрики качества используем и почему они связаны с бизнес-результатом, обозначенным `Product Owner` в разделах `1` и `3`. Пример - WAPE <= 50% для > 80% категорий, bias ~ 0. Возможна формулировка в терминах относительно бейзлайна, количественно. Для бейзлайна могут быть свои целевые метрики, а может их вообще не быть (если это обосновано) `Data Scientist`   
- Необходимый результат этапа. Например, необходимым результатом может быть не просто достижение каких-либо метрик качества, а включение в модели определенных факторов (флаг промо для прогноза выручки, др.) `Data Scientist`    
- Какие могут быть риски и что планируем с этим делать. Например, необходимый для модели фактор (флаг промо) окажется незначимым для большинства моделей. Или для 50% моделей будет недостаточно данных для оценки `Data Scientist`    
- Верхнеуровневые принципы и обоснования для: feature engineering, подбора алгоритма решения, техники кросс-валидации, интерпретации результата (если применимо).  
- Предусмотрена ли бизнес-проверка результата этапа и как будет проводиться `Data Scientist` & `Product Owner`
